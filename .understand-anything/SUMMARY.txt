CORE ARCHITECTURE: 6-LAYER ROUTING CASCADE
================================================================================

The routing engine applies these layers sequentially; first match wins:

LAYER 0: POLICY RULES (confidence 0.95)
  ├─ Declarative rules from config.yaml
  ├─ Match on: model_requested, headers, keywords, tokens, client_profile
  └─ Select with: allow/deny providers, prefer tiers, require capabilities

LAYER 1: STATIC RULES
  ├─ Hardcoded pattern matching
  ├─ Model name aliases and known patterns
  └─ Quick fallback for common cases

LAYER 2: HEURISTICS & SIGNALS
  ├─ Message complexity detection (architecture, performance, reliability, devops, testing, security, database)
  ├─ Device type detection (mobile/tablet)
  ├─ Signal groups: short_message, short_complex, has_tools, knowledge_domain
  ├─ Streaming and cache preferences
  └─ Keyword-based complexity scoring

LAYER 3: REQUEST HOOKS
  ├─ Extensible pre-routing plugins
  ├─ Can modify request body, override profile, inject hints
  ├─ Registered hook functions with async support
  └─ Configurable error handling (fail/continue)

LAYER 4: CLIENT PROFILES
  ├─ Named routing personas: openclaw, n8n, cli, generic
  ├─ Header-based activation (x-faigate-client, x-openclaw-source)
  ├─ Preset select constraints and tier preferences
  └─ Profile-specific defaults and overrides

LAYER 5: LLM CLASSIFIER
  ├─ Optional ML-based classification (disabled by default)
  ├─ Adds latency; recommended only for specialized scenarios
  └─ Future ML-based routing intelligence

FALLBACK: DEFAULT PROVIDER (confidence 0.3)
  ├─ First provider in fallback_chain
  ├─ Guaranteed routing even if all layers fail
  └─ Last resort option

================================================================================
REQUEST HOOKS SYSTEM
================================================================================

Hook Lifecycle:
  1. Registry maintains _REQUEST_HOOKS dict
  2. Config specifies enabled hooks and error handling
  3. Hooks applied in sequence before Layer 3
  4. Results accumulated and merged
  5. Error modes: "fail" (raise) or "continue" (log)

Hook Result:
  body_updates:     dict (messages, model, tools, temperature, max_tokens)
  profile_override: str (override client profile)
  routing_hints:    dict (allow/deny/prefer providers, tiers, capabilities)
  notes:            list (metadata)

Error Handling:
  on_error: "fail"      → Raises HookExecutionError
  on_error: "continue"  → Logs error, proceeds to next layer
